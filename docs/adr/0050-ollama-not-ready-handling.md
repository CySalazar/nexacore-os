# ADR-0050: Ollama "model-not-ready" handling (M1 empty-answer fix)

**Status:** Accepted
**Date:** 2026-06-11
**Deciders:** agent analysis under operator-approved the development plan WS5-10
**Refs:** the development plan WS5-10, ADR-0030 (`OllamaProvider` HTTP seam), ADR-0032 (AI
syscalls → IPC relay), ADR-0035 (M1 `IntentDispatcher`/`RuntimeLink`, LocalCpu
failover), `crates/nexacore-runtime/src/provider/ollama.rs`,
`crates/nexacore-runtime-image/src/remote.rs`, `crates/nexacore-aicheck-image`

## Context

The M1 path (`AiInvoke` → IPC relay → backend router → Ollama `RemoteGpu`)
intermittently returned a **successful but empty** answer on the test VM: the
`nexacore-aicheck` self-test logged `backend_used=RemoteGpu`, `AiInvoke OK,
output_len=0`, `EMPTY answer (transport ok but no content)`, `exit=6`.

Root cause (proven on hardware, 2026-06-11):

- A **tcpdump on `vmbr0`** showed NexaCore sends the CORRECT request
  (`POST /api/generate {"model":"gemma4:latest","prompt":"what is 2+2?",
  "stream":false,"options":{"num_predict":64}}`) and a **warm** model replies
  `{"response":"4",...,"done_reason":"stop"}`. Request, model, prompt and
  transport are all correct.
- A model that is **not resident** (cold, or mid-(un)load) instead replies
  `200` with a ~93-byte body `{"response":"","done":true,"done_reason":"load"
  |"unload"}` — an empty completion, not a real answer.
- Both the Ring-3 client (`remote.rs::GenerateChunk`) and the host provider
  (`provider/ollama.rs::GenerateChunk`) deserialized **only** `response` +
  `eval_count` and **ignored `done`/`done_reason`**, so they returned the
  empty `response` as a *successful* answer (`Ok(("", _))`), which the relay
  forwarded verbatim → `output_len=0`.
- Empirically confirmed: pinning the model resident (`OLLAMA_KEEP_ALIVE=-1` on
  the lab Ollama) makes the same boot return `answer=4` / `M1 OK`. That is a
  mitigation, not a fix — the defect resurfaces whenever a backend is not
  resident (model swap, Ollama restart, OOM eviction, or a Tier-1/2 remote
  backend the device does not control).

## Decisions

1. **Parse `done_reason`.** Add `done_reason: String` (`#[serde(default)]`) to
   both `GenerateChunk`s.

2. **Never return a "successful empty answer".** A reply with an **empty
   `response` AND `done_reason != "stop"`** is treated as *model-not-ready*,
   not as a valid completion:
   - Host (`OllamaProvider::generate`): return `ProviderError::Unavailable`
     (which `is_retriable()` ⇒ the `BackendRouter` retries / fails over).
   - Ring 3 (`remote::generate_once`): return the new `RemoteError::NotReady`.
   - An empty answer **with** `done_reason == "stop"` is a genuine (if rare)
     completion and is passed through unchanged.

3. **Bounded retry before fallback (Ring 3).** `generate` is split into
   `generate_once` (one socket lifecycle) + a `generate` wrapper that, on
   `NotReady`, yields cooperatively (`REMOTE_RETRY_YIELDS`) and re-issues the
   request up to `REMOTE_GENERATE_ATTEMPTS` (6) times — giving the model time
   to finish loading so the user gets the REAL answer. If still not ready, it
   returns `NotReady`, which the relay's existing `Err(_) => local_generate`
   arm routes to the LocalCpu engine (M1 failover, ADR-0035) — a non-empty
   fixture answer, never an empty one.

4. **Build-pipeline hardening (operational).** `deploy-proxmox.sh --remote-build`
   now runs `build-shell-initramfs.sh` before `build-iso.sh`, so the deployed
   ISO always embeds Ring-3 image sources from the current tree (the initramfs
   is `include_bytes!`'d at kernel-compile time and was NOT regenerated by
   `build-iso.sh` alone — a stale-initramfs hazard).

## Consequences

- The M1 empty-answer bug cannot recur: a not-ready backend yields either the
  real answer (after the model loads) or a LocalCpu fallback, never empty.
- The server-side `OLLAMA_KEEP_ALIVE=-1` mitigation becomes optional and can be
  reverted once this lands.
- Host unit tests pin the contract: `done_reason:"load"` empty ⇒ retriable
  `Unavailable`; `done_reason:"stop"` empty ⇒ passed through.
- The Ring-3 retry is time-bounded (6 × `REMOTE_RETRY_YIELDS` cooperative
  yields); a permanently-down backend still terminates to fallback.

## Alternatives considered

- **Fallback-only (no retry):** simplest, but a cold model would always answer
  from the weaker LocalCpu fixture instead of the real GPU model. Rejected in
  favour of retry-then-fallback.
- **Server-side keep-alive only:** masks the symptom in one environment; leaves
  the client defect for every uncontrolled remote backend. Kept only as an
  interim mitigation.
